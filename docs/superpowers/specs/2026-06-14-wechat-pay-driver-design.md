# 微信支付 API v3 驱动设计

## 背景

平台目前只有测试支付：创建订单后调用 `pay-test` 直接完成付款和套餐发放。阶段三需要接入微信支付 API v3，让小程序能够走真实 JSAPI 支付，并通过微信支付通知完成订单入账。

## 目标

- 增加微信支付 API v3 签名、通知验签和通知资源解密能力。
- 增加 JSAPI 预支付下单能力，服务端根据订单金额和用户 `openid` 创建预支付单。
- 增加支付通知入口，核对商户号、订单号、币种、金额和交易状态后入账。
- 保持测试支付可用于本地开发，不提交任何真实商户密钥、证书、API v3 Key 或 AppSecret。

## 非目标

- 本次不实现退款 API v3。
- 本次不在仓库保存真实证书或私钥内容。
- 本次不做真实小额支付验收，真实联调等微信认证和商户号准备好后进行。

## 设计

新增 `apps/api-server/src/payments` 模块：

- `WechatSignatureService`：负责请求签名、通知签名验签、API v3 资源解密。
- `WechatPayClient`：负责 JSAPI 下单请求构造、调用和返回小程序 `wx.requestPayment` 所需参数。
- `WechatPayController`：提供用户发起微信支付和微信支付通知入口。
- `WechatPayService`：编排订单查询、下单、通知入账和幂等处理。

订单模块保持订单和套餐发放的唯一写入边界：

- 创建订单时根据 `PAYMENT_DRIVER` 写入 `TEST` 或 `WECHAT`。
- 测试支付只允许处理 `TEST` 订单。
- 微信支付通知只允许处理 `WECHAT` 订单。
- 已经 `FULFILLED` 的订单再次收到同一微信通知时返回成功，不重复发放套餐。

## 安全要求

- 通知必须先验签，再用 API v3 Key 解密 `resource`。
- 响应和日志不得包含私钥、API v3 Key、AppSecret、完整通知密文或用户隐私输入。
- 金额以服务端订单金额为准，客户端不得传可信金额。
- 通知入账必须核对商户号、订单号、币种、付款金额和交易状态。
- 支付回调入口不需要用户登录，但必须通过微信签名和资源解密校验。

## 测试策略

- 单元测试覆盖签名验签、解密失败、金额不符、重复通知幂等。
- E2E 测试覆盖无效签名返回 401、金额不符返回失败、成功通知发放套餐、重复通知不重复发放。
- 所有测试使用本地生成的测试 RSA 密钥和测试 API v3 Key，不使用真实微信商户资料。

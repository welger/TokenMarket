# 生产就绪检查设计

## 背景

微信小程序已提交认证，正式上线还依赖微信认证、微信支付、HTTPS 域名、真实客服与合规披露等外部事项。等待审核期间，先补一个只读的生产就绪检查，帮助管理员在后台看到哪些事项已经满足、哪些仍会阻塞生产发布。

## 目标

- 后端提供只读生产就绪检查接口，不返回密钥、证书、Cookie、登录态或用户输入内容。
- 管理后台在合规配置页展示检查结果，按 `通过 / 警告 / 阻塞` 告知修复方向。
- 文档沉淀上线清单，方便微信认证通过后逐项验收。

## 非目标

- 本次不接入真实微信支付 API v3。
- 本次不保存微信商户证书、API v3 Key、AppSecret 等真实凭据。
- 本次不启用生产模式，不替用户完成外部账号配置。

## 检查项

生产就绪结果分为三类：

- `PASS`：已满足。
- `WARN`：建议补齐，但不一定阻塞当前开发联调。
- `FAIL`：上线前必须处理。

首批检查项：

- 经营主体与客服：经营主体、客服入口、投诉渠道、服务器地区。
- 数据与协议：日志保存期、业务数据保存期、导出、删除、注销方式，以及隐私政策、用户协议、内容安全规则 URL。
- 供应商披露：至少一个启用供应商填写用途和地区。
- 模型服务：至少一个可用模型绑定启用供应商。
- 内容安全：至少一条启用的内容安全规则。
- 运行环境：生产环境必须关闭测试登录、不能使用测试支付驱动。
- 微信小程序：生产环境必须配置 `WECHAT_APP_ID` 和 `WECHAT_APP_SECRET`。
- 网络代理：生产环境应配置可信反向代理 CIDR。

## API 形状

`GET /admin/compliance/production-readiness`

仅管理员可读，允许 `OWNER`、`OPERATOR`、`SUPPORT`、`AUDITOR` 查看。

```json
{
  "status": "FAIL",
  "generatedAt": "2026-06-13T00:00:00.000Z",
  "summary": {
    "pass": 6,
    "warn": 1,
    "fail": 2
  },
  "checks": [
    {
      "id": "compliance.operator",
      "label": "经营主体",
      "status": "PASS",
      "message": "已填写经营主体"
    }
  ]
}
```

整体状态规则：

- 任一 `FAIL` 时整体为 `FAIL`。
- 无 `FAIL` 且存在 `WARN` 时整体为 `WARN`。
- 全部通过时整体为 `PASS`。

## 安全边界

- 接口只返回是否已配置，不返回配置值本身。
- 密钥、证书路径、AppSecret、API Key、支付密钥不进入响应、不进入日志、不进入文档。
- 检查结果只用于提示和发布门槛，不替代人工核对微信认证、商户号、域名备案和微信后台配置。
